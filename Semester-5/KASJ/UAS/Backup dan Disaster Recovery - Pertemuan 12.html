Mengapa Backup & Disaster Recovery?
Business Impact of Data Loss

Statistik Data Loss

  • 93% perusahaan yang kehilangan data selama 10 hari akan bangkrut dalam setahun
  • 50% perusahaan yang mengalami bencana besar akan tutup dalam 2 tahun
  • $300 billion kerugian global akibat data loss setiap tahunnya

Cyber Threats

Ransomware, malware, cyber attacks yang dapat mengenkripsi atau menghapus data

Hardware Failures

Kerusakan disk, server failure, power outage yang dapat menyebabkan data loss

Human Errors

Kesalahan pengguna, accidental deletion, misconfiguration sistem

Backup Strategies
Types and Approaches

Full Backup

Complete copy of all data

Pro: Complete protection, fast recovery
Con: Time consuming, large storage
Best for: Weekly/monthly backups, critical systems

Incremental Backup

Only changed files since last backup

Pro: Fast backup, less storage
Con: Complex recovery, dependency chain
Best for: Daily backups, large datasets

Differential Backup

Changed files since last full backup

Pro: Faster than full, easier recovery
Con: Growing size, moderate storage
Best for: Balanced approach, medium-size data
3-2-1 Backup Rule
Industry Best Practice
The Golden Rule of Backup
3
Copies

Keep 3 copies of important data

1 original + 2 backups
2
Media Types

Store backups on 2 different media types

Local disk, tape, cloud, etc.
1
Offsite Copy

Keep 1 copy offsite or in cloud

Protection from local disasters

Modern 3-2-1-1 Rule

Add 1 additional copy that is offline and immutable (air-gapped) for ransomware protection

RTO vs RPO
Key Recovery Metrics

RTO

Recovery Time Objective

Berapa lama sistem boleh down sebelum berdampak serius pada bisnis

Example: Email system harus recovery dalam 2 jam

RPO

Recovery Point Objective

Berapa banyak data yang boleh hilang sebelum berdampak serius pada bisnis

Example: Boleh kehilangan maksimal 15 menit data transaksi
Business Criticality RTO Target RPO Target Backup Frequency Cost Impact
Critical < 1 hour < 15 minutes Continuous/Real-time High
Important < 4 hours < 1 hour Hourly Medium
Normal < 24 hours < 8 hours Daily Low
Low Priority < 72 hours < 24 hours Weekly Very Low
Recovery Procedures
Step-by-Step Recovery Process

1. Assess the Situation

Identify the scope of data loss, affected systems, and root cause of the incident

2. Isolate and Secure

Isolate affected systems to prevent further damage and secure the recovery environment

3. Prioritize Recovery

Identify critical systems and data based on business impact and recovery objectives (RTO/RPO)

4. Execute Recovery

Restore data from backups following documented procedures and verify integrity

5. Validate and Test

Test recovered systems, validate data integrity, and ensure full functionality

6. Document and Learn

Document the incident, recovery actions, and lessons learned for future improvements

Setup cron jobs for regular backups 
# Backup Script: backup.sh
#!/bin/bash

SOURCE="/home /etc /var/log"
DESTINATION="/backup/$(date +%Y%m%d)"
REMOTE_HOST="backup-server.local"

# Create backup directory
mkdir -p $DESTINATION

# Perform backup
for dir in $SOURCE; do
    rsync -avz --delete $dir $DESTINATION/
done

# Remote backup
rsync -avz $DESTINATION/ user@$REMOTE_HOST:/backups/

# Add to crontab: 0 2 * * * /scripts/backup.sh

Key Decisions

  • File Server: RTO 8 hours, RPO 4 hours
  • Database: RTO 2 hours, RPO 30 minutes

  • Weekly full backups
  • Monthly offsite backup to cloud
  • Quarterly DR testing

Emergency Actions

  • Notify stakeholders within 30 minutes
  • Setup temporary communication
  • Begin recovery procedures

Data Reconciliation
Recover any missing data

Patch vulnerabilities, update security

Document and improve DR plan
