üöÄ Pengantar Optimasi Kinerja Sistem Terintegrasi

Definisi dan Pentingnya Optimasi

Optimasi kinerja sistem terintegrasi adalah proses berkelanjutan untuk meningkatkan efisiensi, kecepatan, dan skalabilitas sistem dengan mengidentifikasi bottleneck, menerapkan best practices, dan memanfaatkan teknologi yang tepat untuk memaksimalkan value dan meminimalkan resource consumption.

Dalam sistem terintegrasi modern yang kompleks, optimasi tidak hanya tentang membuat sistem "lebih cepat", tetapi tentang menciptakan keseimbangan optimal antara performa, biaya, skalabilitas, dan user experience. Sistem yang optimal adalah sistem yang dapat memberikan layanan terbaik kepada pengguna dengan resource yang efisien.

üéØ Tujuan Optimasi

  • Meningkatkan response time dan throughput
  • Mengurangi latency dan downtime sistem
  • Mengoptimalkan penggunaan resource (CPU, RAM, Disk)
  • Meningkatkan skalabilitas dan capacity planning
  • Mengurangi biaya infrastruktur dan operasional
  • Meningkatkan user experience dan satisfaction

‚ú® Manfaat Optimasi

  • User retention meningkat hingga 70%
  • Conversion rate naik 25-35% untuk setiap 1s faster
  • Biaya cloud infrastructure turun 30-50%
  • SEO ranking meningkat (Core Web Vitals)
  • Competitive advantage di pasar
  • Better developer productivity

üîç Identifikasi Bottleneck dan Masalah Kinerja

Bottleneck adalah titik dalam sistem yang membatasi keseluruhan performa. Mengidentifikasi bottleneck adalah langkah pertama dalam optimasi yang efektif.

Common Performance Bottlenecks

üóÑÔ∏è Database Bottleneck

Gejala: Query lambat, high CPU/memory usage, connection pool exhausted

Penyebab: N+1 queries, missing indexes, unoptimized queries, large table scans

üåê Network Bottleneck

Gejala: High latency, packet loss, timeout errors, slow API calls

Penyebab: Bandwidth limitations, network congestion, geographic distance, DNS issues

‚öôÔ∏è Application Bottleneck

Gejala: Slow processing, high CPU usage, memory leaks, blocking operations

Penyebab: Inefficient algorithms, synchronous blocking, lack of caching, poor code quality

üñ•Ô∏è Infrastructure Bottleneck

Gejala: Resource exhaustion, throttling, disk I/O wait, swap usage

Penyebab: Insufficient resources, poor capacity planning, single point of failure

üîó Integration Bottleneck

Gejala: API timeout, failed integrations, cascade failures, queue backlogs

Penyebab: External API slowness, synchronous dependencies, tight coupling

üë• Concurrency Bottleneck

Gejala: Deadlocks, race conditions, thread starvation, context switching overhead

Penyebab: Poor thread management, shared resources, lock contention

üóÑÔ∏è Teknik Optimasi Database

1. Query Optimization

‚ö†Ô∏è N+1 Query Problem - Common Pitfall

// ‚ùå BAD: N+1 Query Problem
// Fetches 1 query for users, then N queries for each user's posts
users = User.all()  // 1 query
for user in users:
    posts = user.posts.all()  // N queries (1 per user)

// Total: 1 + N queries (e.g., 1 + 1000 = 1001 queries!)

// ‚úÖ GOOD: Eager Loading dengan JOIN
users = User.all().prefetch_related('posts')  // 2 queries total
for user in users:
    posts = user.posts.all()  // No additional query!

// Total: 2 queries only (users + posts)

-- Create index on frequently queried columns

-- Composite index for multiple columns

üéØ Query Optimization Tips

-- ‚ùå BAD: SELECT *
SELECT * FROM users WHERE active = true;

-- ‚úÖ GOOD: Select only needed columns
SELECT id, name, email FROM users WHERE active = true;

-- ‚ùå BAD: Subquery in WHERE
SELECT * FROM orders WHERE user_id IN
  (SELECT id FROM users WHERE country = 'ID');

-- ‚úÖ GOOD: JOIN instead
SELECT o.* FROM orders o
INNER JOIN users u ON o.user_id = u.id
WHERE u.country = 'ID';

üí° Database Performance Best Practices

1. Use indexes wisely - But don't over-index (slows down writes)
2. Avoid SELECT * - Fetch only columns you need
3. Use LIMIT - Pagination untuk large datasets
4. Connection pooling - Reuse database connections
5. Cache query results - Redis/Memcached untuk frequently accessed data
6. Regular VACUUM - Untuk PostgreSQL, clean up dead tuples
7. Monitor slow queries - Log queries > 1 second for analysis

‚ö° Caching Strategies dan CDN

Caching adalah salah satu teknik optimasi paling efektif - dapat meningkatkan performa hingga 10-100x untuk data yang sering diakses.

Multi-Level Caching Architecture

üåê Browser Cache (L1)

Location: Client browser

TTL: Hours to days

Use case: Static assets (JS, CSS, images)

Headers: Cache-Control, ETag

üî• CDN Cache (L2)

Location: Edge servers globally

TTL: Minutes to hours

Use case: Static content, API responses

Providers: CloudFlare, Akamai, CloudFront

üíæ Application Cache (L3)

Location: Application server memory

TTL: Seconds to minutes

Use case: Session data, computed results

Tools: Node-cache, functools.lru_cache

üóÑÔ∏è Distributed Cache (L4)

Location: Dedicated cache servers

TTL: Minutes to hours

Use case: Shared data across instances

üíΩ Database Query Cache (L5)

Location: Database server

TTL: Until data changes

Use case: Frequent identical queries

Built-in: MySQL Query Cache, PgBouncer

üîß Object/ORM Cache (L6)

Location: Application layer

TTL: Session-based or custom

Use case: Database objects, entities

Tools: Django cache, Hibernate 2nd level

üíª Redis Caching Implementation Example

import redis
import json
from functools import wraps

# Connect to Redis
cache = redis.Redis(host='localhost', port=6379, decode_responses=True)

def cache_result(ttl=300):
    """Decorator untuk cache function results"""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            # Create cache key
            cache_key = f"{func.__name__}:{str(args)}:{str(kwargs)}"

            # Try to get from cache
            cached = cache.get(cache_key)
            if cached:
                print(f"‚úÖ Cache HIT: {cache_key}")
                return json.loads(cached)

            # Cache MISS - execute function
            print(f"‚ùå Cache MISS: {cache_key}")
            result = func(*args, **kwargs)

            # Store in cache with TTL
            cache.setex(cache_key, ttl, json.dumps(result))
            return result
        return wrapper
    return decorator

@cache_result(ttl=600)  # Cache for 10 minutes
def get_user_profile(user_id):
    # Expensive database query
    return database.query(f"SELECT * FROM users WHERE id = {user_id}")

# First call - Cache MISS, hits database
profile = get_user_profile(123)

# Second call - Cache HIT, instant response!
profile = get_user_profile(123)

‚ö†Ô∏è Cache Invalidation - "One of the hardest problems in CS"

Strategi Invalidation:

2. Event-based: Invalidate on data update/delete

4. Version-based: Change cache key when data version changes

‚öñÔ∏è Load Balancing dan Horizontal Scaling

Load balancing mendistribusikan traffic ke multiple servers untuk meningkatkan availability, scalability, dan fault tolerance.

Load Balancing Algorithms

üîÑ Round Robin

Cara Kerja: Requests didistribusikan secara bergiliran ke setiap server

Pros: Simple, equal distribution

Cons: Tidak consider server load/capacity

Use case: Homogeneous servers dengan equal capacity

üìä Least Connections

Cara Kerja: Request ke server dengan active connections paling sedikit

Pros: Better untuk long-lived connections

Cons: Slightly more complex

Use case: WebSocket, database connections

‚ö° Least Response Time

Cara Kerja: Request ke server dengan response time tercepat

Pros: Optimal performance

Cons: Requires monitoring

Use case: Mixed workloads

üéØ IP Hash

Cara Kerja: Client IP di-hash untuk determine server

Pros: Session affinity/sticky sessions

Cons: Uneven distribution possible

Use case: Session-based applications

üîß Nginx Load Balancer Configuration

# nginx.conf - Load Balancer Configuration

upstream backend_servers {
    # Load balancing method
    least_conn;  # or: round_robin, ip_hash

    # Backend servers
    server app1.example.com:8001 weight=3;  # Higher weight = more requests
    server app2.example.com:8002 weight=2;
    server app3.example.com:8003 weight=1;
    server app4.example.com:8004 backup;     # Backup server

    # Health checks
    server app5.example.com:8005 max_fails=3 fail_timeout=30s;

    # Keepalive connections
    keepalive 32;
}

server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://backend_servers;

        # Headers
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

        # Timeouts
        proxy_connect_timeout 5s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;

        # Load balancing improvements
        proxy_next_upstream error timeout invalid_header http_500 http_502 http_503;
        proxy_next_upstream_tries 2;
    }

    # Health check endpoint
    location /health {
        access_log off;
        return 200 "healthy\n";
    }
}

‚úÖ Horizontal Scaling Best Practices

1. Stateless applications - Store session di Redis/database, bukan memory

üíª Code Optimization Best Practices

Algorithm & Data Structure Optimization

‚ö° Big O Complexity - Choose Wisely!

// ‚ùå BAD: O(n¬≤) - Nested loops
function findDuplicates(arr) {
    let duplicates = [];
    for (let i = 0; i < arr.length; i++) {
        for (let j = i + 1; j < arr.length; j++) {
            if (arr[i] === arr[j]) {
                duplicates.push(arr[i]);
            }
        }
    }
    return duplicates;
}

// ‚úÖ GOOD: O(n) - Hash Set
function findDuplicates(arr) {
    let seen = new Set();
    let duplicates = new Set();
    for (let item of arr) {
        if (seen.has(item)) {
            duplicates.add(item);
        }
        seen.add(item);
    }
    return Array.from(duplicates);
}

// Performance: 1000 items = 1,000,000 ops vs 1,000 ops (1000x faster!)

// Memoization - Cache expensive calculations
const memoize = (fn) => {
  const cache = {};
  return (...args) => {
    const key = JSON.stringify(args);
    if (cache[key]) return cache[key];
    const result = fn(...args);
    cache[key] = result;
    return result;
  };
};

// Expensive fibonacci calculation
const fibonacci = memoize((n) => {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
});

// fibonacci(50) - instant with memoization!

‚ö†Ô∏è Common Performance Anti-Patterns

1. Premature Optimization: "Optimize what matters" - Profile first!

üîß Custom Performance Monitoring Implementation

// Performance monitoring decorator
class PerformanceMonitor {
    constructor() {
        this.metrics = [];
    }

    // Measure function execution time
    measure(name, fn) {
        return async (...args) => {
            const startTime = performance.now();
            const startMemory = process.memoryUsage().heapUsed;

            try {
                const result = await fn(...args);
                const endTime = performance.now();
                const endMemory = process.memoryUsage().heapUsed;

                this.recordMetric({
                    name,
                    duration: endTime - startTime,
                    memory: endMemory - startMemory,
                    timestamp: new Date(),
                    status: 'success'
                });

                return result;
            } catch (error) {
                const endTime = performance.now();

                this.recordMetric({
                    name,
                    duration: endTime - startTime,
                    timestamp: new Date(),
                    status: 'error',
                    error: error.message
                });

                throw error;
            }
        };
    }

    recordMetric(metric) {
        this.metrics.push(metric);

        // Alert if slow
        if (metric.duration > 1000) {
            console.warn(`‚ö†Ô∏è Slow operation: ${metric.name} took ${metric.duration}ms`);
        }
    }

    // Get performance summary
    getSummary(functionName) {
        const funcMetrics = this.metrics.filter(m => m.name === functionName);
        const durations = funcMetrics.map(m => m.duration);

        return {
            count: funcMetrics.length,
            avgDuration: durations.reduce((a, b) => a + b, 0) / durations.length,
            minDuration: Math.min(...durations),
            maxDuration: Math.max(...durations),
            p95: this.percentile(durations, 95),
            p99: this.percentile(durations, 99)
        };
    }

    percentile(arr, p) {
        const sorted = arr.sort((a, b) => a - b);
        const index = Math.ceil((p / 100) * sorted.length) - 1;
        return sorted[index];
    }
}

// Usage
const monitor = new PerformanceMonitor();

const getUserData = monitor.measure('getUserData', async (userId) => {
    const user = await db.users.findById(userId);
    const orders = await db.orders.findByUser(userId);
    return { user, orders };
});

// Analyze performance
console.log(monitor.getSummary('getUserData'));
// { count: 100, avgDuration: 45ms, p95: 120ms, p99: 250ms }

        'api status 200': (r) => r.status === 200,
        'api response valid': (r) => r.json().length > 0,
    });

    // Record metrics

    // Test 3: POST request

        name: 'Test User',
        email: 'test@example.com'
    });

    response = http.post('https://api.example.com/users', payload, {

‚ùì Common Questions

Q: Kapan mulai optimize? Awal development atau setelah production?
A: Best practice: Design untuk performance dari awal (choose right architecture), tapi detailed optimization setelah identify actual bottlenecks dengan real data.

Q: Tools mana yang paling recommended untuk pemula?
A: Start dengan Chrome DevTools (free, built-in), Lighthouse, dan WebPageTest. Upgrade ke APM tools (New Relic, Datadog) untuk production systems.

Q: Berapa target ideal untuk page load time?
A: General rule: <2s untuk desktop, <3s untuk mobile. Tapi depends on industry - e-commerce perlu faster (<1s) untuk better conversion.

Q: Apakah optimization sekali jalan atau continuous?
A: Continuous! Performance degrades over time (more features, more data). Need regular monitoring & periodic optimization sprints.
